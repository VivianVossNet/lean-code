# Манифест Lean Code

## Преамбула

Clean Code решал реальную проблему в 2008 году. Java-энтерпрайз-монолиты содержали методы по 3 000 строк без какой-либо структуры. Рецепт был обоснован.

Индустрия скопировала его как универсальный закон.

Lean Code — не противоположность Clean Code. Это то, чем Clean Code должен был стать: минимальный набор принципов, масштабируемый от одной функции до целой системы.

Никаких правил на количество строк. Никакой догмы извлечения. Никаких церемоний.
KISS в голове, а не правила на стене.

---

## Аксиома

**Сохраняй когнитивный ресурс разработчика для логики.**

Когнитивный ресурс конечен. Каждая лишняя интерпретация, каждое переключение контекста, каждый прыжок в другой файл — это ресурс, который не идёт на понимание логики.

Все принципы этого манифеста служат одной цели: держать голову разработчика свободной для настоящей работы.

---

## I. Структура

Как код нарезается.

### I.1 Один юнит — одна задача

**I.1.A** Функция делает одно. Файл несёт одну ответственность. Модуль служит одной цели.

**Почему:** Голова мгновенно следит за одной понятной вещью.

### I.2 Область задачи определяет размер

**I.2.A** Область задачи определяет размер юнита. Не количество строк. Тридцать читаемых строк для одной задачи — это lean. Четыре строки для фрагмента, вырванного из контекста, — нет.

**I.2.B** Вопрос никогда не «сколько строк?», а «сколько задач?»

**Почему:** Голова теряет контекст, когда приходится прыгать.

---

## II. Именование

Как код обозначается.

### II.1 Глагол и область из закрытого словаря

**II.1.A** Имя функции следует паттерну `{verb}{scopeInMaxThreeWords}`. Глагол берётся из конечного набора:

| Verb | Meaning |
|------|---------|
| `read` | Retrieve data |
| `write` | Store data |
| `create` | Bring into existence |
| `delete` | Remove from existence |
| `update` | Modify existing data |
| `find` | Search by criteria |
| `check` | Verify a condition |
| `parse` | Transform a format |
| `render` | Produce output |

**II.1.B** Область описывает домен максимум в трёх словах. Вместе глагол и область читаются как короткое предложение. Если нужно больше трёх слов — область слишком широка. Раздели задачу.

`readOrderFromDatabase`. `checkStockAvailability`. `createInvoiceFromOrder`. `writeInvoiceToFile`. `parseConfigFromYaml`. Комментарий не нужен. Имя — это контракт.

**Почему:** Один глагол — одно значение — везде. Интерпретация не требуется.

### II.2 Номенклатура

**II.2.A** Пиши полностью. `readConfiguration`, а не `readConfig`. Имя — это предложение. Предложения не содержат сокращений.

**II.2.B** Выбирай самое простое слово. `read`, а не `fetch`. `check`, а не `validate`. `write`, а не `persist`. Чем проще слово, тем меньше интерпретации оно требует.

**II.2.C** Никаких синонимов. Никаких `handle` или `process`, за которыми прячется реальное действие. Один глагол — одно значение — везде.

**Почему:** Каждый перевод в голове потребляет ресурс.

### II.3 Пространство имён несёт область, имя несёт существительное

**II.3.A** Структуры данных несут существительное. Пространство имён несёт область. `billing.Order`, а не `BillingOrder`. `stock.Item`, а не `StockItem`. Пространство имён делает одну работу. Имя типа — другую. Ни одно не делает работу другого.

**II.3.B** Следуй соглашениям об именовании того языка, на котором пишешь. `read_order_from_database` в Python. `readOrderFromDatabase` в TypeScript. `ReadOrderFromDatabase` в Go. Принцип остаётся. Регистр адаптируется.

**Почему:** Каждая часть имени выполняет одну задачу. Это принцип I.1, применённый к именованию.

## III. Область видимости

Как код остаётся разделённым.

### III.1 Понятен там, где читается

**III.1.A** Код читают гораздо чаще, чем пишут. Если нужно прыгать между восемью файлами, чтобы понять одну операцию, — структура провалилась. Если нужно держать граф вызовов в голове, чтобы проследить логику, — абстракция провалилась.

**III.1.B** Lean-код понятен там, где он читается.

**Почему:** Каждое переключение контекста стоит когнитивного ресурса.

### III.2 Одинаковый код, разная область — остаётся раздельным

**III.2.A** Две функции могут содержать идентичный код. Если они обслуживают разные обязанности, которые могут меняться независимо, — они остаются раздельными. Их объединение связывает две области. Когда одна изменится — абстракция сломается.

**III.2.B** Дублирование — не враг. Преждевременное связывание — враг.

**Почему:** Связанные обязанности заставляют думать обо всём одновременно.

### III.3 Комментарии

**III.3.A** Файл начинается с заголовка: лицензия и область файла. Это единственный комментарий, существующий по соглашению.

**III.3.B** Всё остальное самодокументируется. Если коду нужен комментарий, чтобы быть понятым, — нарушен принцип. Исправь имя (II), исправь структуру (I) или исправь область (III). Не исправляй комментарием.

**Почему:** Комментарий — это признание того, что код не говорит сам за себя.

## Фрактал

Эти принципы — не семь отдельных правил. Это одно правило, применённое рекурсивно:

**Каждый юнит делает одно, размер определяется областью, имя берётся из закрытого словаря, читается на месте и разделён по обязанностям.**

Это верно для функции. Это верно для файла. Это верно для модуля. Это верно для системы.

Когда кодовая база растёт, принципы не меняются. Они экстраполируются. Система масштабируется, потому что правила фрактальны, а не потому что добавляются новые правила.

---

## Чем Lean Code не является

**Не грязный код.** Lean-код структурирован, именован и осознан.

**Не оправдание для длинных функций.** Функция на 500 строк, делающая двенадцать вещей, нарушает I.1.

**Не отрицание абстракций.** Абстракции — инструменты. Используй их, когда они снижают сложность. Не когда они просто перемещают её.

**Не догма.** Если ситуация требует нарушить принцип — нарушь. Затем задокументируй почему.

---

## Проверка

Открой любой файл. Прочитай любую функцию. Если ты понимаешь, что она делает, не прыгая в другое место, — это lean.

Если не понимаешь — нет.
