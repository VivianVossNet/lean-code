# 精益代码宣言

## 前言

Clean Code 在 2008 年解决了一个真实的问题。Java 企业级单体应用里到处是三千行的方法，毫无结构可言。那套药方确实对症。

但整个行业把它当成了普适法则来照搬。

精益代码不是 Clean Code 的反面。它是 Clean Code 本该成为的样子：一组最小化的原则，能从单个函数扩展到整个系统。

没有行数限制。没有提取教条。没有形式主义。
KISS 装在脑子里，不是贴在墙上的规矩。

---

## 公理

**把开发者的认知容量留给逻辑本身。**

认知容量是有限的。每一次多余的解读、每一次上下文切换、每一次跳转到另一个文件，都是没有花在理解逻辑上的容量。

本宣言中的所有原则只服务于一个目的：让开发者的大脑专注于真正的工作。

---

## I. 结构

代码如何切分。

### I.1 一个单元，一件事

**I.1.A** 一个函数做一件事。一个文件承担一个职责。一个模块服务于一个目的。

**为什么：** 你的大脑能立刻跟上一件清晰的事。

### I.2 职责决定大小

**I.2.A** 职责的范围决定单元的大小，而不是行数。三十行可读的代码完成一件事，是精益的。四行代码把一个片段从上下文中硬生生拽出来，不是。

**I.2.B** 问题从来不是"多少行？"而是"多少件事？"

**为什么：** 大脑在跳转时会丢失上下文。

---

## II. 命名

代码如何标注。

### II.1 动词与作用域来自封闭词汇表

**II.1.A** 函数名遵循 `{动词}{最多三个词的作用域}` 的模式。动词来自一个有限集合：

| Verb | Meaning |
|------|---------|
| `read` | Retrieve data |
| `write` | Store data |
| `create` | Bring into existence |
| `delete` | Remove from existence |
| `update` | Modify existing data |
| `find` | Search by criteria |
| `check` | Verify a condition |
| `parse` | Transform a format |
| `render` | Produce output |

**II.1.B** 作用域用最多三个词描述所属领域。动词加作用域读起来像一个短句。如果你需要超过三个词，说明作用域太宽——拆分任务。

`readOrderFromDatabase`。`checkStockAvailability`。`createInvoiceFromOrder`。`writeInvoiceToFile`。`parseConfigFromYaml`。不需要注释。名字就是契约。

**为什么：** 一个动词，一个含义，处处如此。无需解读。

### II.2 命名规范

**II.2.A** 完整拼写。`readConfiguration`，不是 `readConfig`。名字是一个句子。句子里不包含缩写。

**II.2.B** 选最朴素的词。`read`，不是 `fetch`。`check`，不是 `validate`。`write`，不是 `persist`。词越简单，需要的解读就越少。

**II.2.C** 不用同义词。不用含糊的 `handle` 或 `process` 来掩盖实际发生的事。一个动词，一个含义，处处如此。

**为什么：** 脑中的每一次翻译都在消耗认知容量。

### II.3 命名空间承载作用域，类型名承载名词

**II.3.A** 数据结构承载名词。命名空间承载作用域。`billing.Order`，不是 `BillingOrder`。`stock.Item`，不是 `StockItem`。命名空间做一件事。类型名做另一件事。互不越权。

**II.3.B** 遵循你所使用语言的命名惯例。Python 里写 `read_order_from_database`。TypeScript 里写 `readOrderFromDatabase`。Go 里写 `ReadOrderFromDatabase`。原则不变。大小写随语言适配。

**为什么：** 名字的每一部分各司其职。这是原则 I.1 在命名上的应用。

---

## III. 作用域

代码如何保持隔离。

### III.1 在阅读处即可理解

**III.1.A** 代码被阅读的次数远多于被编写的次数。如果你需要在八个文件之间跳转才能理解一个操作，说明结构失败了。如果你需要在脑中保持一张调用图才能跟上逻辑，说明抽象失败了。

**III.1.B** 精益代码在阅读处即可理解。

**为什么：** 每一次上下文切换都有认知成本。

### III.2 相同代码，不同职责，保持独立

**III.2.A** 两个函数可能包含相同的代码。如果它们服务于不同的关注点，且这些关注点可以独立变化，就让它们保持独立。合并它们会耦合两个作用域。当其中一个变化时，抽象就会崩塌。

**III.2.B** 重复不是敌人。过早耦合才是。

**为什么：** 耦合的关注点迫使你同时思考多件事。

### III.3 注释

**III.3.A** 文件以头部信息开头：许可证和文件的作用域。这是唯一按惯例存在的注释。

**III.3.B** 其余一切靠代码自解释。如果代码需要注释才能被理解，说明某条原则被违反了。修正命名（II），修正结构（I），或修正作用域（III）。不要用注释来修补。

**为什么：** 注释是对代码无法自我表达的承认。

---

## 分形

这些原则不是七条独立的规则。它们是一条规则的递归应用：

**每个单元只做一件事，大小由职责决定，命名来自封闭词汇表，在原地即可读懂，按关注点隔离。**

这对函数成立。对文件成立。对模块成立。对系统成立。

当代码库增长时，原则不会改变。它们向外推演。系统之所以能扩展，是因为规则是分形的，而不是因为添加了新规则。

---

## 精益代码不是什么

**不是脏代码。** 精益代码是有结构、有命名、有意图的。

**不是长函数的借口。** 一个五百行做了十二件事的函数违反了 I.1。

**不是反抽象。** 抽象是工具。当它能降低复杂度时使用。不是当它仅仅转移复杂度时。

**不是教条。** 如果某种情况需要打破原则，就打破它。然后记录原因。

---

## 检验标准

打开任意文件。阅读任意函数。如果你不需要跳转到别处就能理解它做了什么，它就是精益的。

如果不能，它就不是。
